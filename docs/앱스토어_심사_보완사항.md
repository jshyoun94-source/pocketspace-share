## 1. Guideline 2.3.8 – 앱 아이콘 (Placeholder 아이콘)

**거부 사유:** 앱 아이콘이 최종본이 아니거나 placeholder로 보인다. 아이콘이 서로 비슷하고 일관되어 사용자가 앱을 찾을 수 있어야 한다.

**보완:**
- **iOS:** `ios/PocketSpace/Images.xcassets/AppIcon.appiconset/` 에 들어가는 모든 아이콘을 **최종 디자인**으로 교체 (1024x1024 포함, 각 해상도별 동일 디자인).
- **Expo/공통:** `assets/images/icon.png` 를 최종 앱 아이콘 이미지로 교체 후 `npx expo prebuild --clean` 또는 Xcode에서 앱 아이콘 에셋만 다시 설정.
- 플레이스홀더·테스트용 이미지가 하나라도 남아 있지 않도록 확인.

---

## 2. Guideline 2.3.3 – iPad 스크린샷

**거부 사유:** 13인치 iPad 스크린샷이 iPhone 이미지를 늘리거나 수정한 것처럼 보인다. 스크린샷은 앱의 핵심 개념과 기능을 보여줘야 하며, 각 지원 기기에서의 실제 사용 모습을 정확히 반영해야 한다.

**보완:**
- App Store Connect → 해당 앱 → **미리보기 및 스크린샷** → **미디어 관리자에서 모든 크기 보기**.
- **iPad용 스크린샷**을 **iPad 시뮬레이터 또는 실제 iPad**에서 앱을 실행한 뒤 캡처한 이미지로 **새로 업로드** (iPhone 화면을 iPad 크기로 늘린 이미지 사용 금지).
- iPhone용·iPad용을 각각 해당 기기 화면 비율에 맞게 구분해서 등록.

---

## 3. Guideline 4.8 – 로그인 (동등한 로그인 옵션)

**거부 사유:** 서드파티 로그인(Google 등)을 사용하는 앱은, 아래를 모두 만족하는 **다른 로그인 옵션 하나**를 제공해야 한다.
- 이름·이메일 정도로만 데이터 수집 제한
- 계정 설정 시 이메일을 비공개로 할 수 있는 옵션
- 동의 없이 앱 내 행동을 광고 목적으로 수집하지 않음  

Apple은 **Sign in with Apple**이 위 조건을 만족한다고 명시하고 있음.

**보완:**
- **Sign in with Apple**을 로그인 화면에 추가하는 것이 가장 확실함.
- 구현 시: Expo/React Native에서는 `expo-apple-authentication` 등으로 Apple 로그인 연동 후, Firebase Auth와 연동(Apple ID 토큰으로 커스텀 토큰 발급 등).
- 이미 다른 로그인(예: 카카오/네이버)이 위 조건을 만족한다고 판단되면, App Store Connect **심사 팀에 대한 답글**에서 “어떤 로그인이 위 조건을 만족하는지, 어떻게 만족하는지”를 명확히 설명해야 함. 다만 Apple은 보통 Sign in with Apple 추가를 권장함.

---

## 4. Guideline 2.1 – 앱 완성도 (버그)

**거부 사유:** Google 로그인 후 앱이 반응하지 않음 (iPad Air 11인치 (M3), iPadOS 16.2.1).

**보완:**
- **iPad**에서 실제 또는 시뮬레이터로 Google 로그인 플로우를 끝까지 테스트.
- 로그인 성공 후 화면 전환(리다이렉트/딥링크 처리), 로딩 상태, 에러 처리 등에서 멈추거나 흰 화면이 나오는지 확인.
- Google OAuth 리다이렉트 URL·스킴이 iPad에서도 동일하게 동작하는지 확인 (예: `com.jshyoun94.pocketspace` 스킴).
- 수정 후 **지원 기기 목록에 있는 모든 기기·OS 버전**에서 한 번씩 로그인 테스트 권장.

---

## 5. Guideline 1.5 – 지원 URL

**거부 사유:** App Store Connect에 입력한 지원 URL(`https://tangential-sled-352810.web.app`)이 사용자가 문의하고 지원을 요청할 수 있는 정보를 제공하는 페이지로 연결되지 않는다.

**보완:**
- 해당 URL(또는 Support URL에 넣을 URL)이 열리면 **문의 방법·지원 요청 방법**이 명확히 보이도록 수정.
- 예: “고객 지원 / 문의” 섹션에 **이메일 주소**, **문의/지원 요청 방법 안내** (필요 시 간단한 문의 양식 또는 연락처)를 노출.
- 현재 사업자 정보·서비스 소개만 있고 “문의/지원 요청” 안내가 없으면 거부 사유에 해당하므로, **반드시 지원/문의 안내**를 추가했음 (아래 `public/index.html` 수정 반영).

## GitHub 업로드 및 빌드·제출 순서 (main 브랜치 기준)

1. **로컬에서 보완 작업 반영 후**
   - `git status` 로 변경 파일 확인  
   - `git add .` (또는 필요한 파일만)  
   - `git commit -m "앱스토어 심사 보완: 지원 URL 문의 안내 추가 등"`  
   - `git push origin main`

2. **iOS 빌드**
   - Xcode: `ios/PocketSpace.xcworkspace` 열기 → Product → Archive → Distribute App → App Store Connect 업로드  
   - 또는 EAS Build 사용 시: `eas build --platform ios --profile production` 후 같은 계정으로 업로드

3. **App Store Connect**
   - 빌드가 처리된 후 “빌드” 선택  
   - 미리보기/스크린샷(특히 iPad) 재업로드  
   - 지원 URL이 문의·지원 안내 페이지로 연결되는지 최종 확인  
   - “심사 제출” 클릭

4. **Sign in with Apple / 로그인 관련**
   - 구현했다면: 앱 내 로그인 화면과 App Store Connect 심사 메모에 “Sign in with Apple 제공” 명시  
   - 구현하지 않고 기존 로그인으로 답변할 경우: “심사 팀에 대한 답글”에 어떤 로그인이 4.8 조건을 만족하는지 구체적으로 작성
